# Skill.md 概念介绍

## 什么是Skill.md？

Skill.md是一种给AI提供"工作手册"的标准化文档格式。就像入职新公司时拿到的《开发规范文档》，Skill.md告诉AI：

- **项目如何组织代码** - 目录结构、文件命名
- **有哪些约定俗成的规矩** - 编码规范、命名习惯
- **常见业务怎么处理** - 表单、CRUD、错误处理
- **代码写成什么样才算合格** - 质量标准、测试要求

**核心价值：一次配置，持续复用。**

不用每次都重复说"我用React 18"、"组件要写在components文件夹"，AI读了Skill.md就全懂了。

## Skill.md vs Prompt的区别

| 对比维度 | Prompt | Skill.md |
|----------|--------|----------|
| 使用时机 | 每次对话都要写 | 写一次，长期使用 |
| 内容性质 | 具体任务描述 | 通用规范和知识 |
| 适用场景 | "帮我生成一个登录页" | "我们项目所有页面的开发规范" |
| 更新频率 | 随需求变化 | 技术栈变更时才改 |
| 典型长度 | 100-500字 | 1000-5000字 |
| 维护成本 | 高（每次都要写） | 低（一次编写） |
| 复用性 | 低 | 高（跨项目可用） |

## Skill.md的核心组成部分

### 1. 技术栈约定
```markdown
## 技术栈约定
- React 18 + TypeScript
- Ant Design 5.x
- React Router v6
- React Query
- Zustand
```

### 2. 项目结构规范
```markdown
## 文件结构规范
src/
├── components/          # 公共组件
├── pages/              # 页面组件
├── hooks/              # 自定义Hooks
└── utils/              # 工具函数
```

### 3. 代码规范
```markdown
## 命名规范
- 组件文件：PascalCase.tsx
- 变量命名：camelCase
- 常量命名：UPPER_SNAKE_CASE
```

### 4. 业务模板
```markdown
## CRUD模板
每个CRUD模块包含：
- 列表页（搜索、分页、表格）
- 详情页（信息展示）
- 编辑页（表单验证）
```

### 5. 输出要求
```markdown
## 输出要求
AI生成代码时必须：
1. 遵循所有规范
2. 包含完整类型定义
3. 添加错误处理
4. 提供测试用例
```

## Skill.md的工作原理

### AI如何读取Skill.md？
当AI接收到用户请求时：

1. **识别Skill需求**：用户提到"按照xxx规范"
2. **查找Skill文件**：在项目目录中寻找对应的.skill.md文件
3. **读取并理解**：解析Skill文件的结构和要求
4. **应用规范生成代码**：按照Skill中的约定生成代码
5. **自我检查**：确保生成的代码符合Skill要求

### 示例：生成表单组件
```
用户：请按照 form-generator.skill.md 规范，生成用户注册表单

AI步骤：
1. 读取 form-generator.skill.md
2. 了解需要：React + TypeScript + Ant Design + react-hook-form + Zod
3. 按照文件结构创建：UserRegisterForm/index.tsx, schema.ts, types.ts
4. 根据规范添加字段：用户名、邮箱、密码
5. 添加校验规则：必填、格式、长度
6. 添加提交逻辑：loading状态、错误处理
7. 输出完整代码
```

## 为什么需要Skill.md？

### 解决的问题
1. **重复沟通**：不用每次告诉AI技术栈和规范
2. **代码不一致**：不同AI或不同时间生成的代码风格统一
3. **质量参差**：确保AI生成的代码达到团队标准
4. **新人上手慢**：新成员通过Skill.md快速了解项目

### 带来的好处
1. **效率提升**：AI生成代码更准确，减少返工
2. **质量保证**：代码符合团队标准
3. **知识沉淀**：团队经验固化到文档
4. **协作顺畅**：所有人使用同一套标准

## Skill.md的适用场景

### 1. 个人开发者
- 快速启动新项目
- 保持代码风格一致
- 复用成功经验

### 2. 开发团队
- 统一团队编码规范
- 新成员快速上手
- 代码审查有据可依

### 3. 开源项目
- 贡献者指南
- 代码规范说明
- 开发流程说明

### 4. 教育培训
- 教学模板
- 练习规范
- 评分标准

## Skill.md的局限性

### 1. 学习成本
- 需要学习Skill.md的编写格式
- 需要花时间整理团队规范

### 2. 维护成本
- 技术栈更新时需要更新Skill.md
- 新业务模式需要新增Skill.md

### 3. 适用边界
- 不适合一次性任务
- 不适合探索性项目
- 不适合原型验证

## 最佳实践

### 1. 渐进式采用
1. 从一个小的Skill开始（如表单生成）
2. 验证效果，收集反馈
3. 逐步扩展到其他领域

### 2. 版本控制
- Skill.md也纳入git管理
- 记录更新日志
- 支持回滚

### 3. 团队协作
- 定期评审Skill.md
- 收集使用反馈
- 持续改进

### 4. 文档配套
- 提供使用示例
- 编写操作指南
- 维护常见问题
